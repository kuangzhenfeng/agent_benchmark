# Q1：SubscriptionHub 组合缺陷修复

`SubscriptionHub` 是进程内事件总线。最近的压测同时打开了延迟 drain、回调重入和 32 位时钟回绕，出现偶发 payload 损坏、卡死、重复投递和永不过期的订阅。基础冒烟测试通常通过。

修复 `include/subscription_hub.hpp` 和 `src/subscription_hub.cpp`，不改变任何公开方法签名，也不引入 C++17 以外的依赖。

## 运行方式

```bash
./run_public_checks.sh
```

公开检查包含基础路径和 callable 析构重入路径；starter 能通过基础路径，但完整公开检查预期失败。请自行补充其他边界语义的测试。

## 必须满足的语义

1. `subscribe` 和 `publish` 的 `std::string_view` 输入可能来自临时 `std::string`。任何跨调用保存的数据必须拥有自己的存储；回调拿到的 `Event` 视图在该次回调返回前有效。
2. 回调可同步调用 `subscribe`、`unsubscribe`、`publish`、`drain` 和 `expire_idle`，且不得死锁、使用失效引用、漏清理队列或无限递归处理当前批次。
3. 一次 `drain(now_tick)` 只处理调用开始时已经在队列中的事件。回调中新发布的事件必须留给下一次 drain；每个开始投递时仍处于订阅状态的订阅者至多收到一次该事件。
4. 回调中取消订阅必须阻止该订阅者收到当前事件之后的事件；回调中新建的订阅不接收本次 drain 已截取的事件。
5. 单个回调抛异常不能阻止其余符合条件的回调。该事件对已经尝试调用的订阅者仍视为处理完成，不能在下一次 drain 被重复投递；`DispatchStats::delivered` 统计所有尝试调用次数（包括抛异常的回调），`callback_errors` 单独计数异常。
6. `expire_idle(now_tick, ttl)` 使用 32 位模运算判断经过时间；在 `ttl < 2^31` 的前提下，经过时间大于或等于 TTL 即过期。首次订阅与每次实际尝试回调后更新活跃 tick。TTL 为 0 时立即过期。
7. 公共方法可被多个线程调用。内部锁不可在用户回调或用户可执行代码期间持有；实现不应把整个 drain 串行为一把长时间全局锁。`Callback` 及其保存的用户 callable 的复制、移动和销毁（包括释放最后一个所有者时触发的析构）都视为可能执行用户代码，必须在内部锁外完成。

## 约束与验收

- 不得改变 `Event`、`DispatchStats` 或 `SubscriptionHub` 的公共 API，也不得通过忽略异常或删除并发支持规避问题。
- **硬性封顶条件（该题总分不超过 60）**：违反第 7 条中 callable 生命周期的锁边界要求。
- 可以调整私有字段、增加私有类型或新增公开检查。
- 需要在 `ANSWER.md` 中说明事件批次快照、取消订阅和异常之间的处理顺序。
- 验收会组合临时字符串、回调重入（含 callable 析构重入）、取消/新增订阅、抛异常、跨 `uint32_t` 回绕和多线程调用；仅修复单一症状不视为完成。

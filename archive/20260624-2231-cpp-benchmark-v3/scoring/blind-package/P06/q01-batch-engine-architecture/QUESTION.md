# Q1：为 TaskForge 引擎绘制 5 种风格的 SVG 架构图

`TaskForge` 是一套用 C++17 写成的进程内任务调度 / 批处理引擎，源码就在本目录的 `include/` 与 `src/` 下，可编译运行（运行 `./run_public_checks.sh` 会编译并跑一个 demo，打印运行时统计）。

请**通读源码**，理解它的架构后，在 `diagrams/` 目录下产出 **5 张 SVG 格式的架构图**，分别覆盖以下 5 种风格：

| 文件名 | 风格（`data-style` 取值） | 视角 |
|--------|---------------------------|------|
| `01_layered_architecture.svg` | `layered` | 分层架构图 |
| `02_component_dependency.svg` | `component` | 组件依赖图 |
| `03_dataflow.svg` | `dataflow` | 数据流图 |
| `04_class_relationship.svg` | `class` | 类与模块关系图 |
| `05_deployment_topology.svg` | `deployment` | 部署 / 运行时拓扑图 |

本题是**产物型任务**：你要读懂代码，把架构"画"出来，而不是改代码。评分会对你产出的 5 张 SVG 盲评。

## 运行方式

```bash
./run_public_checks.sh
```

它会做两件事：① 编译并运行引擎 demo，让你能观测真实运行时行为（worker 线程数、重试、成功/失败统计）；② 用 `svg_check.py` 校验你产出的 5 张 SVG 是否合法且符合标注约定。

> 初始状态下，5 张 SVG 是占位图，`svg_check.py` 会判定**失败**（缺 `data-style` 与标注容器）。这是预期状态，不是对你能力的预判。你的目标是把它们替换为带标注的真实架构图，使公开检查通过。

## SVG 标注约定（关键，决定正确性如何被核对）

你的配色、布局、形状、字体、装饰**完全自由**，但为了让架构事实可被核对，每张 SVG 必须按下列约定嵌入语义结构（用 SVG 原生支持的 `data-*` 自定义属性）：

```xml
<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 1200 800"
     data-style="layered">                       <!-- 取值见上表 -->
  <g data-role="nodes">
    <g data-id="Scheduler" data-kind="class">     <!-- 一个语义节点 -->
      <!-- 任意图形 + <text>Scheduler</text> -->
    </g>
    <!-- ... 更多节点 ... -->
  </g>
  <g data-role="edges">
    <line data-from="Scheduler" data-to="TaskQueue" data-kind="depends"/>
    <!-- 边也可以用 <path>/<polyline>，但必须有 data-from / data-to / data-kind -->
  </g>
</svg>
```

约定要点：

1. 根 `<svg>` 必须有 `data-style`，取值为该图的风格（`layered` / `component` / `dataflow` / `class` / `deployment`），且必须有 `viewBox` 或同时有 `width`+`height`。
2. 节点放在 `data-role="nodes"` 容器内，每个节点用 `data-id` 标明其对应源码中的模块/类名（建议用源码里的英文标识符，如 `Scheduler`、`WorkerPool`、`ITaskStore`；评分时对中英文与大小写做了归一化）。
3. 边放在 `data-role="edges"` 容器内，每条边用 `data-from` / `data-to` 指向节点 id，并用 `data-kind` 标明关系类型。
4. 各风格的 `data-kind` 词表：
   - `layered`：`depends`
   - `component`：`depends`（编译期依赖）、`callback`（仅运行时回调）
   - `dataflow`：`flow`（有向数据流，配合 `<text>` 标注流转内容更佳）
   - `class`：`inherits` / `implements` / `composes` / `aggregates` / `uses` / `depends`
   - `deployment`：`contains` / `connects` / `runs-on` / `callback`
5. 公开检查要求：每张图至少 6 个节点、4 条边，且 `data-style` 合法。

评分时，机器脚本会解析你的 `data-*` 标注，与私有 ground truth 做集合匹配，计算各视角的节点/边覆盖率与陷阱命中度——**因此标注是否准确、是否对得上源码，是"正确性"维度的主要依据**，与图的"好看程度"相对独立。

## 你需要从源码中提炼的架构事实

不同风格的图关注不同的架构事实，请逐张据源码准确表达：

- **分层图**：按真实编译期依赖方向分层（应用入口 / 编排 / 领域服务 / 基础设施）。注意：分层应依据**真实依赖关系**，而非源码注释的自称（例如某组件注释自称"核心"，但按依赖方向它其实属于基础设施层）。
- **组件依赖图**：模块间的编译期依赖方向。特别留意：哪些是真正的编译期依赖，哪些只是运行时通过注入的回调产生的联系——两者不要混淆，更不要把运行时回调画成编译期循环依赖。
- **数据流图**：一次任务从提交、入队、被 worker 取出执行、产出结果、失败重试、到终态通知的完整流转（含重试反馈环）。
- **类关系图**：接口与实现的继承/实现关系、类的组合/聚合、以及"依赖抽象接口还是依赖具体类"。组合根（main）装配了哪些具体实现，运行时实际实例化的是哪些。
- **部署图**：运行时的实例与线程、数量（如 worker 线程数）、共享的汇合点（队列）、实例间的归属与连接关系；以及哪些类虽编译进产物但运行时并未被实例化。

## 提示：源码里存在若干会"一眼看错"的陷阱

下列都不是脑筋急转弯，都能从源码推导，但容易画错；准确处理它们是本题的主要区分点：

- **命名相近但语义不同的类型**：存在一个活跃的优先级队列和一个签名相似、注释自称"通用"但全仓库无人调用的旧队列——分清哪个是真正在用的。
- **死代码 / 历史遗留**：存在定义完整但无人调用的类或方法——它们是否应进入核心架构，或应如何标注。
- **运行时回调 vs 编译期依赖**：某模块通过运行时注入的回调"反向"调用了上层——这是运行时回调，不是编译期循环依赖，画图时要正确区分。
- **注释自称与真实归属不符**：某组件注释自称"核心调度"，但按依赖方向其实是基础设施层叶子节点。
- **接口依赖 vs 具体类依赖**：上层持有的是抽象接口指针，具体实现由组合根装配——依赖边应画到接口，而非具体实现。

不需要"猜"任何题面之外的信息；以上每一点都可以在 `include/` 与 `src/` 中找到确切证据。

- 5 张 SVG 必须各自能被解析（well-formed XML）、可渲染，且符合标注约定（`data-style` 合法、有 nodes/edges 容器、节点 ≥6、边 ≥4）。

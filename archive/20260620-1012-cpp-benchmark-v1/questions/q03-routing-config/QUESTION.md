# Q3：RoutingConfig 安全发布重构

网关使用 `RoutingConfig` 热更新路由。当前实现把可变配置、观察者通知和兼容 API 混在同一把锁中。线上发生过 reader 读到半更新配置、回调重入死锁、观察者异常阻断 reload，以及 `current()` 返回引用在 reload 后悬空的问题。

请重构 `include/routing_config.hpp` 与 `src/routing_config.cpp`，在 C++17 下保持现有公开 API 的源代码兼容性。重点是清楚的生命周期与并发语义，而不是最少改动行数。

## 运行方式

```bash
./run_public_checks.sh
```

公开检查包含基础 reload、跨实例 `current()` 生命周期和 observer 复制重入路径；starter 能通过基础路径，但完整公开检查预期失败。请为本题新增其他边界测试。

## 必须满足的语义

1. `reload(candidate)` 必须先完整校验，再一次性发布新配置。校验失败（空 prefix、空 target、重复 prefix）时返回 `false`，当前快照、版本和观察者均不变。
2. 成功 reload 后版本严格加一。`snapshot()` 必须返回不可变的、完整自洽的配置快照；并发 reader 可以无锁或短锁读取，但不能看到部分更新，也不能得到悬空对象。
3. 保持 `const Config& current() const` 的签名。它返回的是该实例当前不可变快照的引用；在该调用线程对**同一 `RoutingConfig` 实例**下一次调用 `current()` 前、以及之后的任意 reload 中，引用必须保持有效且仍指向该次快照。该线程调用另一 `RoutingConfig` 实例的 `current()` 不得影响前一实例引用的生命周期；实现应在题目约束下说明其生命周期策略。
4. 每次成功发布后，调用开始时仍已订阅的每个观察者至多通知一次，收到刚发布的不可变快照。通知不得在内部互斥锁下执行；观察者可重入地 `subscribe`、`unsubscribe`、`reload` 或读取快照。`Observer` 及其保存的用户 callable 的复制、移动和销毁（包括释放最后一个所有者时触发的析构）同样视为可能执行用户代码，必须在内部锁外完成。
5. 观察者抛异常不能阻止其他观察者或回滚已发布配置；`observer_error_count()` 累加每一次抛出的异常。回调中取消订阅只影响后续 reload，回调中新建订阅不接收当前通知。
6. `find_target` 必须针对传入路径返回最长匹配 prefix 的 target；无匹配返回空字符串。该查询必须只基于一个一致快照。
7. 所有公共方法可由多个线程并发调用。不得通过永久持有锁、全局静态可变配置或泄漏快照来规避生命周期问题。

## 约束与验收

- 不得删改现有公开方法、公开结构体字段或方法签名。可添加私有辅助类型、私有字段和测试。
- 只使用 C++17 标准库；不要依赖平台级 RCU 库或线程局部的全局共享可变路由表。
- 在 `ANSWER.md` 中说明发布点、`current()` 的引用保活机制、观察者快照时刻与异常隔离。
- 验收会并发执行 reload、snapshot、find_target 和重入观察者，检查原子发布、跨实例引用生命周期、版本、最长前缀匹配、通知次数和 observer callable 的特殊成员重入。

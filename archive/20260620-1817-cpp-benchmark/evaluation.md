## 总结

本轮 5 个参评对象在 v2 长上下文任务上**整体正确率极高（Q2 全员 100%、Q1 4/5 全对）**，差距主要体现在**验证证据（ANSWER.md）与个别字段误判**，而非实现能力。**claude code + glm-5.2 与 opencode + glm-5.2 并列领先**（两题真值 100% + 验证证据最详尽）；codex + gpt-5.5 紧随其后（两题全对、证据完整但略简）；claude code + qwen3.7-max 因 Q1 误判 4 字段排第四；opencode + qwen3.7-max 虽 Q1 仅错 3 字段、Q2 全对，但**两题 ANSWER.md 均未填写**，验证证据维度严重失分，排末位。结论置信度中等（正确率客观可验，验证维度为评分员判断）。

- 匿名 ID：P01/P02/P03/P04/P05（按 slug 原序分配，仅用于解盲）

## 关键差异

- **实现正确率高度趋同**：5 个对象派生函数（accepts/replayable/flag_active/phase_rank/branch_of/within_band）字节级等价且正确；Q2 全员 480/480。这是同题 + 同 starter 的正常结果，本轮 v2 在"实现质量"维度区分度低，区分主要来自**字段误判与文档**。
- **字段误判（Q1）**：仅 claude code + qwen3.7-max（4 字段）与 opencode + qwen3.7-max（3 字段）有真实错误，集中于 route_code/phase_value/mode_flag 的跨来源取舍（errata vs proto vs changelog 优先级边界）。
- **验证证据（ANSWER.md）是主要拉分点**：
  - claude code + glm-5.2、opencode + glm-5.2：两题 ANSWER 极详尽（逐字段来源 / ERR 全量核对），验证满分。
  - opencode + qwen3.7-max：**两题 ANSWER.md 均为空模板**，尽管源码真值正确率仍高（Q1 99.4%、Q2 100%），验证维度仅得 1-2 分，拖累排名至末位。
- **耗时**：codex + gpt-5.5 最快（~11m）且两题全对；opencode + glm-5.2、claude code + glm-5.2 用时最长（~31-36m）但产出验证证据最详尽。

## 风险与限制

- **样本量**：N=5，单一预设（v2 长上下文/协议适配），2 题。不应外推到 bugfix、并发、重构等其他题型。
- **区分度偏低**：本任务正确率全员 ≥99%，差距集中在 ANSWER.md（主观）与个别字段；"领先"幅度（199 vs 195）未达"明显领先"阈值（平均分差≥15）。按阈值应表述为**"本轮未拉开显著差距"**，但存在基于验证证据的微弱排序。
- **未运行的部分**：所有提交均通过公开检查（退出码 0，主 agent 已复核）；私有 ground truth 全量核对由确定性脚本完成，客观可信。
- **编译器**：本机 g++ 9.4.0，C++17 支持充分；各参评对象运行环境编译器版本未逐一固定（记录为限制）。
- **时间盒未强制**：推荐 300 分钟/题，实际均在 ~10-36 分钟内完成；超时控制（7200s 后台）未触发。
- **ANSWER.md 空白**：opencode + qwen3.7-max 两题未填 ANSWER.md，无法验证其取舍过程；源码真值正确说明其内部推导正确，但过程证据缺失。

## 下一轮建议

- **提高区分度**：若要拉开模型间差距，可引入更高陷阱密度（更多 errata 推翻/近名变体）或要求**逐字段 provenance 必填**（把 ANSWER.md 纳入硬性约束），避免"源码对但无文档"的情况。
- **补测题型**：v2 仅测长上下文/协议适配；建议搭配 v1（bugfix/实现/重构）综合排名，避免单一题型偏置。
- **关注 opencode + qwen3.7-max**：其实现正确率不低但文档习惯缺失，可在工程协作维度单列观察。

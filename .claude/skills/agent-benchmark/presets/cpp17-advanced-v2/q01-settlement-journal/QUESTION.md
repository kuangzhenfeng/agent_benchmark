# Q1：结算账本协议缺陷修复

支付平台接收商户的授权、扣款、撤销和退款命令。起始实现能处理部分基础路径，但把业务协议简化成了错误的通用支付直觉：跨商户消息混淆、失败请求被错误记忆、调用方文本被借用，以及审计回调在锁内执行。

修复 include/settlement_journal.hpp 与 src/settlement_journal.cpp，不得修改公开 API，也不得引入 C++17 以外的依赖。

## 业务协议

业务协议不使用标记文本文档描述。业务协议/结算账本协议.hpp 是 1,000 行的 C++ 结构体协议，逐项给出业务情形、标识作用域、状态优先级、金额边界、文本所有权和回调边界。本文件只提供导航；结构体协议与本题面共同构成公开合同。

## 核心规则

1. 订单主键为 merchant_id 与 order_id；成功幂等主键为 merchant_id 与 message_id。标识是不透明且区分大小写的字节串。
2. merchant_id 与 message_id 非空后，必须先查成功幂等记录。命中后返回 duplicate，并复用首次 applied 回执的状态字段；忽略这次的订单、金额、版本和命令种类，不审计。
3. 新请求被拒绝时不得写幂等记录；以后可用相同 message_id 重试。新订单只接受 revision 为 1 的正金额 authorize。
4. 已有订单先检查 revision 是否恰为当前版本加一；不相等返回 revision_gap。版本正确后才判断撤销终态和金额规则。
5. capture 必须为正且不超过授权未扣余额；refund 必须为正且不超过已扣未退余额；void_authorization 必须为零金额且从未扣款。
6. 成功 void 后，版本正确的新命令返回 already_voided；所有失败与重复调用均不得改变订单。
7. 跨调用保存的文本必须拥有副本。AuditRecord 也必须拥有文本。
8. 仅新成功命令在状态提交后审计一次。审计回调的调用、复制、移动、析构和异常处理均不得发生在内部锁持有期间；回调可重入所有公开方法。

## 约束与验收

- 只使用 C++17 标准库；可替换私有实现和增加公开测试。
- **硬性封顶条件（该题总分不超过 60）**：AuditSink 或其用户 callable 的调用、复制、移动或析构发生在内部锁持有期间。
- ANSWER.md 必须解释协议优先级、幂等存储边界、文本所有权和审计锁边界。
- 验收会组合临时字符串、跨商户相同 message_id、篡改重复负载、失败后重试、版本与撤销优先级、审计重入/异常和并发调用。

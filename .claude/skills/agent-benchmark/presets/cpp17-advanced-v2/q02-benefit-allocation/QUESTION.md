# Q2：权益分配协议实现

营销平台需要根据活动预算、客户额度、领取回放和批次原子性发放权益。现有代码只是起始骨架；请实现 include/benefit_allocation.hpp 对应的 src/benefit_allocation.cpp。不得修改公开 API 或枚举，只使用 C++17 标准库。

## 业务协议

业务协议/权益分配业务协议.json 是 3,000 行的 JSON 业务协议。它以活动、客户、领取和批次的组合情形编码业务时钟、额度优先级、单条回放、批次临时账本、回滚和审计规则。本文件只提供导航；JSON 协议与本题面共同构成公开合同。

## 核心规则

1. 单条成功回放键为 campaign_id 与 claim_id；客户用量键为 campaign_id 与 customer_id；批次回放键仅为 batch_id。所有标识不透明且区分大小写。
2. 单条调用先检查非空活动/领取标识，再查成功回放。命中 duplicate 时复用首次 granted 的数值字段，不检查本次客户、金额或业务时刻，也不审计。
3. 单条失败不保存回放。窗口为 starts_at 包含、ends_at 不包含；now 由调用方传入。
4. 单条金额必须为正。客户额度与活动余额同时不足时，customer_limit 优先；否则才是 campaign_exhausted。成功时余额、客户累计和成功回放一次性提交。
5. 非空 batch_id 在任何行校验前查询批次回放。成功和拒绝批次都保存；重复批次的 code 为 duplicate，回复内容复用历史批次。
6. 新批次按输入顺序对临时账本模拟。历史成功 claim 是 batch_conflict；当前批次中重复的 claim 是 batch_duplicate_claim。首个失败只返回一个模拟失败回复，前序临时扣减全部回滚。
7. 所有行成功时才原子提交整个批次，并将每行写入成功单条回放。视图只能看到已提交状态。
8. 审计仅覆盖新成功的单条和成功批次行，使用拥有文本的记录，回调可重入且不在内部锁内执行。

## 约束与验收

- 可以替换全部私有实现和增加公开测试；禁止忙等、轮询 sleep 或在用户回调期间持锁。
- **硬性封顶条件（该题总分不超过 60）**：批次部分扣减被并发调用观察到，或 AuditSink 用户 callable 在内部锁持有期间执行、复制、移动或析构。
- ANSWER.md 必须解释单条与批次回放差异、临时账本、回滚、提交点和审计边界。
- 验收会组合窗口边界、跨活动批次、双额度不足、单条失败重试、篡改重复负载、拒绝批次回放、临时字符串和审计重入。

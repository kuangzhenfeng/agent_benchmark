# Q3：协议一变更向协议二适配

结算平台已在协议一完成一次结构契约升级，现在需要把这次业务变更适配到相似但不等同的协议二。请仅修改 src/protocol_adaptation.cpp，使其规则表严格匹配协议二的二十四张实现绑定卡片。

## 公开输入

- 协议一/结算适配源协议.proto：5,000 行 Protobuf 源业务协议，记录协议一完成后的结构契约。
- 协议一/协议变更映射.json：协议一变更与协议二目标卡片的映射。
- 协议二/结算适配目标协议.json：5,000 行 JSON 目标业务协议；每张卡片分别给出目标字段。
- include/protocol_adaptation.hpp：不可修改的公开 API。
- src/protocol_adaptation.cpp：仍保存迁移前规则的起始实现。

## 适配范围

仅适配协议二中“是否实现绑定”为 true 的二十四张卡片。它们分散在 249 张结构卡片中，且名称具有单复数、视图、补丁等近似形式；每个字段必须独立读取。非绑定卡片只提供上下游业务上下文，不得新增为枚举或规则项。

协议二的字段与 C++ 枚举对应如下：

| JSON 字段 | JSON 值 | C++ 值 |
|-----------|----------|--------|
| 标识策略 | 必填 | IdentifierPolicy::required |
| 标识策略 | 保留空值 | IdentifierPolicy::preserved_if_empty |
| 标识策略 | 空值生成 | IdentifierPolicy::generated_if_empty |
| 回放域 | 无回放 | ReplayDomain::none |
| 回放域 | 账户内回放 | ReplayDomain::account |
| 回放域 | 商户内回放 | ReplayDomain::merchant |
| 回放域 | 批次内回放 | ReplayDomain::batch |

accepts 必须继续由规则表驱动：仅“必填”拒绝空标识；“需要源时间”和“需要序号”互相独立；replayable 仅在回放域不是“无回放”时返回真。

## 约束与验收

- 不得修改公开头文件、测试、协议一、协议二或结构体集合。
- 不得根据名字、相邻位置、协议一卡片或常见支付惯例推断协议二字段。
- 不得对不透明标识或回放令牌增加 trim、哈希、负载一致性比较或全局去重。
- 公开检查只抽样；评分会检查全部二十四项和范围外修改。
- **硬性封顶条件（该题总分不超过 60）**：把协议一的完成值机械复制到协议二，或修改了任一非绑定结构、公开 API、测试或协议工件。

运行方式：

    ./run_public_checks.sh
